# SceneBlueprint 实现与迭代文档

> 版本：v0.1
> 日期：2026-03-28
> 状态：执行中

---

## 1. 文档定位

本文档用于描述 SceneBlueprint 的实现策略、迭代节奏、验证方式和阶段性交付物。

本文档与《架构设计文档》分离：

- 架构设计文档：描述长期设计边界、模块职责、数据边界与总体原则
- 实现与迭代文档：描述近期如何落地、如何验证、如何快速试错和收敛

本文档不试图定义最终架构，也不固化尚未稳定的实现细节。

---

## 2. 当前实施原则

### 2.1 快速迭代优先

SceneBlueprint 当前阶段采用快速迭代策略，不追求在编码前把所有设计一次性定稿。

当前原则：

- 先建立最小可运行闭环，再逐轮补强
- 优先暴露真实交互问题，而不是只在抽象层讨论
- 文档跟随实现收敛，但不提前锁死细节
- 每轮迭代只解决少量核心不确定性

### 2.2 可视化优先验证

外部编辑器不是末期包装层，而是核心产品的一部分。

因此当前阶段要尽快搭出可视化界面，用来持续验证：

- 交互流程是否顺畅
- 数据边界是否合理
- 编辑模型是否过于复杂或耦合
- 导出链路是否容易理解和调试
- Timeline、Graph、Inspector、白模预览之间是否存在断裂

### 2.3 测试与性能观测同步进入

自测、性能测试和人工 walkthrough 不作为后补工作，而作为每轮迭代的基础设施同步建设。

当前原则：

- 有 UI 原型时，就要有基础交互自测
- 有导出链路时，就要有导出结果校验
- 有复杂状态更新时，就要有性能观测
- 没有自动化覆盖的关键流程，至少要有人工检查清单

---

## 3. 当前阶段目标

当前阶段的目标不是“功能齐全”，而是尽快形成一个可持续迭代的骨架应用。

第一阶段更关注以下能力：

- 能启动并展示基础外部编辑器界面
- 能加载基础定义数据
- 能创建、编辑、保存最小蓝图数据
- 能执行最小导出链路
- 能通过自测验证关键流程
- 能记录基础性能数据

换句话说，当前优先交付的是：

**一个能看、能点、能保存、能导出、能测试、能观察性能的最小骨架。**

---

## 4. 第一批实施重点

### 4.1 可视化界面骨架

第一批 UI 不追求完整美术或复杂功能，但必须尽快形成稳定工作台。

建议优先具备这些区域：

- 主工作区
- 节点图区域
- Inspector 区域
- Timeline 区域占位
- 日志 / 调试信息区域
- 顶部工具栏或命令入口

界面骨架的目标不是好看，而是尽快支撑真实编辑流程。

### 4.2 最小数据闭环

第一批实现必须尽快打通以下闭环：

- 读取定义数据
- 新建或打开 `.blueprint.json`
- 在 UI 中进行最小编辑
- 保存编辑结果
- 触发导出
- 得到最小 runtime contract

在闭环未打通前，不宜过早投入大量高级功能开发。

### 4.3 自测体系

第一批自测至少覆盖三类内容：

- 逻辑测试
  - schema / descriptor 读取
  - 导出结果稳定性
  - 命令与状态更新正确性
- 交互自测
  - 基本打开、编辑、保存、导出流程
  - 关键交互不崩溃、不丢状态
- 回归测试
  - 已修复的问题不在后续迭代中反复出现

### 4.4 性能基线

第一批性能工作不追求极致优化，而是尽快建立可比较的基线。

建议尽快开始记录：

- 应用启动时间
- 大图加载时间
- 节点拖拽时的交互流畅度
- Inspector 更新耗时
- Timeline 滚动和缩放耗时
- 导出耗时

当前阶段最重要的是“能持续观测”，不是“立即最优”。

## 5. 每轮迭代的标准流程

后续迭代建议采用固定循环：

1. 选择一个小范围目标
2. 实现最小可工作的功能
3. 进行人工交互 walkthrough
4. 补充或更新自动化测试
5. 记录性能数据
6. 根据结果修正文档、边界或实现

每轮结束时，至少要能回答：

- 这轮新增能力是否真的提升了编辑体验
- 是否暴露了新的边界问题
- 是否引入了新的状态复杂度
- 是否出现了明显性能退化
- 是否需要更新实现文档或架构文档

---

## 6. 当前推荐的验证方式

### 6.1 人工 walkthrough

每轮都应进行真实操作验证，而不是只看测试通过。

建议重点检查：

- 创建蓝图是否顺手
- 节点选择、拖拽、连线是否自然
- 属性编辑是否容易理解
- 保存与导出是否让人困惑
- 错误信息是否足够可读
- Timeline 与节点图之间是否容易来回切换

### 6.2 自动化测试

自动化测试优先覆盖稳定边界，不追求一开始覆盖所有 UI 细节。

建议优先覆盖：

- descriptor / schema 解析
- `.blueprint.json` 读写一致性
- export 结果校验
- 命令式编辑流程
- 关键状态转换

### 6.3 性能观测

性能问题不能等功能完成后再处理。

建议从第一阶段起保留：

- 基础耗时日志
- 可比较的性能样本
- 典型场景的基准数据
- 大图 / 高频交互下的简单观测结果

---

## 7. 当前阶段不优先的事项

为了保证迭代速度，当前阶段不优先追求：

- 完整功能覆盖
- 完整概念穷举
- 过度抽象的底层框架
- 过早锁死最终数据结构
- 复杂的多端同步机制
- 过早优化不影响当前迭代速度的细节

这些内容不是不做，而是要等最小闭环稳定后再逐步推进。

---

## 8. 文档维护原则

本文档应持续更新，但保持偏实施、偏迭代、偏验证。

更新时优先记录：

- 当前阶段目标
- 当前迭代重点
- 当前验证方式
- 已暴露的问题类型
- 已形成共识的实施策略

本文档不负责沉淀长期边界定义；长期边界仍以架构设计文档为准。

---

## 9. 当前结论

SceneBlueprint 当前阶段采用以下落地策略：

- 尽快搭出可视化界面
- 尽快打通最小编辑闭环
- 自测体系同步进入
- 性能基线同步进入
- 通过真实交互持续检验内部逻辑与边界设计

这意味着后续每次迭代都应同时关注：

- 用户交互是否顺畅
- 内部状态是否合理
- 导出链路是否稳定
- 性能是否开始退化
